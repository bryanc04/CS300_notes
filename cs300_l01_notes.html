Memory Model & Addressing

high yield

The Mailbox Model

  • Memory = array of byte-sized boxes; each stores exactly one byte (0–255)
  • Each box has a unique address — a number from 0 to M−1
  • On an 8 GB machine: M = 8 × 230 = 233 addresses
  • On a 16 GB machine: M = 16 × 230 = 234 addresses

Why Addresses Need 8 Bytes

  • 4 bytes → max value ≈ 4 billion → insufficient for >4 GB RAM
  • This limitation actually hit in mid-2000s as RAM exceeded 4 GB
  • Modern CPUs use 8 bytes per address (supports quintillions)
  • Hardware grows address size in powers of two → jumped directly from 4 to 8
ℹ Gibibyte vs Gigabyte: 1 GiB = 230 bytes ≈ 1.07 × 109 bytes. 1 GB (SI) = exactly 109 bytes. The "-bi-" suffix (kibi, mebi, gibi) signals base-2. Exams may test whether you know 230 ≠ 109.

Number Systems: Binary & Hexadecimal

high yield

Positional Notation

In any base b, digit at position i (from right, 0-indexed) contributes digit × bi.

Binary:  1012  = 1×2² + 0×2¹ + 1×2⁰ = 510
Hex:     8D16  = 8×16¹ + 13×16⁰ = 14110
                       ^D = 13

Hex ↔ Binary Shortcut

Exactly 4 binary digits = 1 hex digit (because 24 = 16). This lets you convert without going through decimal:

1001 1101 00102
 9    D    2      → 9D216

This is why hex is preferred: every byte = exactly 2 hex digits.

Pointers in C

high yield

Core Operators

OperatorNameMeaning
&xaddress-ofReturns the memory address of variable x
*pdereferenceFollows pointer p to access the value it points to

Key identity: *(&x) == x — the operators are inverses.

Type Rules

  • int* = "pointer to int" — occupies 8 bytes, points to 4 bytes
  • char* = "pointer to char" — occupies 8 bytes, points to 1 byte
  • *(int*) cancels to int — the star "cancels" the star in the type
  • void* = untyped pointer (what malloc returns)
void f() {
    int  local = 1;
    int* ptr   = &local;   // ptr holds the address of local (8 bytes)

    printf("%d\n",  *ptr);  // dereferences ptr → prints 1
    printf("%p\n", ptr);   // prints address itself (%p = pointer format)
    printf("%p\n", &local); // same address as ptr

    *ptr = 42;             // modifies local through the pointer
    printf("%d\n", local); // → 42
}
⚠ Trap: int* a, b; declares a as int* but b as plain int, not int*. The asterisk binds to the variable name, not the type keyword. Prefer int *a, *b; to make this clear.

Stack vs Heap — Key Differences

  • Stack: contiguous, no gaps. LIFO. Freed automatically when function returns.
  • Heap: can have "holes" (fragmentation) when objects in the middle are freed.
  • Stack grows downward; heap grows upward — they meet if you run out of memory.
  • Java's new → heap. Primitive types (int, double) in Java → stack.

malloc / free Rules

  • malloc(n) returns void* — you must cast it
  • Returns address of the first byte of allocated block
  • You must call free(ptr) with the original pointer
  • Memory leak = malloc without free
  • Double free = free the same pointer twice → undefined behavior
  • Use-after-free = accessing memory after free → undefined behavior
⚠ Trap — object lifetime mismatch: Returning a pointer to a local variable is undefined behavior — the local's memory is reclaimed when the function returns. Only heap-allocated (dynamic lifetime) objects can safely outlive the function that creates them.

Strings & the NUL Terminator

high yield

C String Representation

  • A string is simply a sequence of char bytes in memory
  • Every string ends with a NUL byte: \0 = 0x00
  • A 12-character string occupies 13 bytes — the NUL counts!
  • char[] vs char*: array stores chars; pointer stores an address

ASCII Key Facts

  • Original ASCII: 7 bits → 128 characters
  • 'A' = 65 = 0x41
  • 'a' = 97 = 0x61 (lowercase = uppercase + 32)
  • '0' = 48 = 0x30 (digit char ≠ integer value)
  • NUL (\0) = 0; separate from digit '0' = 48
  • Today: Unicode (1–4 bytes/char); backward-compatible with ASCII
// "Hello" in memory — 6 bytes, not 5!
char s[] = "Hello";
// Address:  0  1  2  3  4  5
// Value:   'H''e''l''l''o''\0'
//           72 101 108 108 111  0

// Heap-allocated string — COMMON TRAPS:
char* p = malloc(6);       // need 6 bytes for "Hello" + NUL
*p = "Hello";              // WRONG — assigns a char from the address, not the string
p  = "Hello";              // WRONG — overwrites pointer; heap memory is lost (memory leak)
strcpy(p, "Hello");       // CORRECT way to copy a string into allocated memory
⚠ Segfault cause: Reading past a string's NUL byte accesses unowned memory. If you go far enough (e.g., 20,000 bytes) the OS terminates the program with SIGSEGV (segmentation fault) — it detected access outside a valid memory segment.
Forgetting the NUL terminator is one of the most common bugs in C. When allocating space for a string of n characters, always allocate n+1 bytes.

char* vs char[] — Critical Distinction

char* s = "Sedan"char s[] = "Sedan"
Data lives inRead-only .rodataStack (copied there)
Modifiable?NO — segfaultYes
Variable size8B (pointer to .rodata)6B (actual chars + NUL)
Safe to return?Yes — .rodata persists foreverNO — stack frame dies on return

String literals go to .rodata for efficiency (string pooling — one copy shared), persistence (survives function return), and safety (OS enforces read-only). char s[] = "hello" uses .rodata as a template but copies the bytes onto the stack.

Pointer Array ≠ Contiguous String Data

When you have char* strs[] = {s1, s2}, the pointers are contiguous but the strings they point to are not:

Stack: strs[0](8B) | strs[1](8B)  ← pointers ARE adjacent
              ↓              ↓
Heap:  "cat\0"         "dog\0"    ← strings NOT adjacent
       0x1a0050        0x1a0080   ← gap between them!

s1[3] = ' ';  // removes NUL from "cat"
printf("%s", strs[0]);
// Prints "cat" then reads garbage — NOT "dog"
// "dog" is not at s1+4!
07

Programs as Bytes

conceptual

Key Insight

The CPU doesn't distinguish between "program bytes" and "data bytes" — it interprets whatever bytes are at the address it's told to execute. The same 4 bytes (0x8d 0x04 0x37 0xc3) can be found inside a JPEG image and executed as x86-64 machine code that adds two numbers.

  • Machine code for a simple add(a, b) function compiles to just 4 bytes
  • lea (%rdi, %rsi, 1), %eax = add the two arguments; retq = return
  • The meaning of bytes in memory depends entirely on how the program chooses to interpret them
  • objdump -d file.o disassembles: shows hex bytes on left, human-readable assembly on right

Compilation Pipeline

  • gcc -o add add.c addf.c compiles both files separately into .o files, then links
  • A declaration = tells compiler a function/variable exists (no implementation)
  • A definition = provides the actual implementation/storage
  • You can use something that's declared but not yet defined — the linker resolves it later

Interpretation Depends on Context

  • Same bytes = machine code or pixel data or an integer — depends on how you access them
  • The computer has no inherent notion of "type" — that's enforced by the program / compiler
  • const unsigned char add[] = {0x8d, 0x04, 0x37, 0xc3}; stores raw bytes; the CPU will execute them if told to jump there

Uninitialized Memory

high yield

What malloc() returns

  • Contents of freshly malloc()'d memory are completely undefined by the C standard
  • In practice: memory often retains whatever bytes were there before — old strings, passwords, etc.
  • This is a major security hole: freed memory can be re-allocated to a different caller who then sees old secrets
  • calloc(n, size) zero-initializes; malloc() does not

Secure Cleanup Pattern

  • Before calling free() on sensitive data, overwrite with zeros first
  • This is standard in cryptographic and password-handling code
  • Global/static variables in the .bss segment are zero-initialized by the OS at startup — local and heap variables are not
⚠ Trap: Partially-initialized structs are dangerous. Members you didn't explicitly set may hold garbage. Use memset(&s, 0, sizeof(s)) or a full initializer list to be safe.

Arrays & Pointer Arithmetic

high yield

Array Fundamentals

  • All elements same type, same size, laid out consecutively in memory
  • Array length is NOT stored in memory — you must track it yourself
  • Array length is fixed at declaration; you cannot append
  • int a[5] → 20 bytes on the stack (5 × 4)
  • sizeof(arr) = total bytes (e.g., 12 for int[3]); but sizeof(ptr) = 8 always

Pointer Arithmetic Rules

  • ptr + 1 advances by sizeof(*ptr) bytes, not 1 byte
  • (int*)p + 1 → address + 4 bytes
  • (char*)p + 1 → address + 1 byte
  • a[i] is exactly *((int*)a + i) — subscript notation is syntactic sugar for pointer arithmetic
  • First member rule: address of array = address of a[0]
int a[3] = {10, 20, 30};

// These four are ALL equivalent:
a[1]                 // = 20
*( a + 1 )           // = 20  (a decays to int* pointer)
*((int*)a + 1)       // = 20  (explicit cast)
1[a]                 // = 20  (legal but cursed: i[a] == a[i] in C!)

// sizeof behavior — critical exam distinction:
sizeof(a)            // = 12  (whole array: 3 × 4 bytes)
int* p = a;
sizeof(p)            // = 8   (just a pointer — NOT the array size)
⚠ Trap — array decay: When you pass an array to a function, it decays to a pointer to its first element. Inside the function, sizeof(arr) gives 8 (pointer size), not the array size. Always pass the length separately.

Structs, sizeof & typedef

high yield

Struct Access Syntax

  • s.member — when s is a value (stack-allocated struct)
  • p->member — when p is a pointer to a struct
  • p->member is identical to (*p).member — arrow implicitly dereferences
  • Address of a struct = address of its first member (first member rule)

sizeof Traps

  • sizeof is resolved entirely at compile time — zero runtime cost
  • sizeof(int*) = 8 — not the size of what it points to
  • sizeof(arr) = total bytes only when arr is an array type, not a pointer
  • sizeof(struct x_t) ≥ sum of member sizes due to padding
  • Use alignof(T) to get alignment (same compile-time mechanism)
// typedef removes the need to write "struct" everywhere
typedef struct {
    int  i1, i2, i3;
    char c1, c2, c3;
} x_t;

x_t s;              // no "struct" keyword needed
s.i1 = 42;

x_t* p = (x_t*)malloc(sizeof(x_t));
p->i1 = 42;        // arrow: pointer to struct
(*p).i1 = 42;      // identical, but uglier

// Linked list: must use "struct node*" not "node_t*" inside definition
typedef struct node {
    int         value;
    struct node* next;  // node_t* would be illegal here — not yet defined
} node_t;
⚠ Self-referential struct trap: You cannot embed a node_t inside itself (infinite size), but you can embed a struct node* pointer (always 8 bytes). The typedef name isn't available until after the closing };, which is why you need the raw struct node* syntax inside the definition.
ℹ Struct initializer shorthand: struct x_t s = {1, 2, 3, 'A', 'B', 'C'}; fills in order. struct x_t s2 = { .i2 = 42, .c3 = 'X' }; sets only named members — unset members get zero if the struct is global/static, garbage otherwise.

The Five Collection Rules

  1. First member rule: address of collection = address of first member
  2. Array rule: elements laid out consecutively, no gaps
  3. Struct rule: members in declaration order, no overlap, minimum padding for alignment
  4. Minimum rule: memory used shall be minimum possible without breaking other rules

⚠ Struct padding rules: Compiler inserts padding before a member if needed for alignment, and may add trailing padding after the last member so that an array of this struct keeps alignment for the next element. You cannot reorder struct members to save space — but you can reorder them yourself in source. Stack variables can be reordered by the optimizer; struct members cannot.
Compute struct size: walk members in order, insert padding before each to satisfy its alignment, then add trailing padding to make total size a multiple of the largest member's alignment (not just the largest member's size — alignment = size for primitives). This trailing padding rule exists so arrays of the struct keep every element's members correctly aligned.

Trailing Padding Trap — Changing int* to int Doesn't Shrink the Struct

A common exam question: if you replace an 8-byte pointer member with a 4-byte int, why is the struct still the same size?

struct costume_info {
    char  name[17];      // ends at 17
    // 1B padding (align short to 2)
    short contestantNum; // ends at 20
    int   score;         // ends at 24
    long  costume;       // ends at 32  ← largest alignment = 8
    int*  studentId;     // ends at 40  ← total 40B ✓ (mult of 8)
};
// Change int* → int: raw data ends at 36B
// 36 is NOT a multiple of 8 → compiler adds 4B trailing padding → still 40B!
// WHY? If you made an array of this struct, the 'long costume' in the
// second element must start at a multiple of 8. 36+4=40 ensures this.
ℹ The rule: struct total size must be a multiple of its largest member's alignment. Even if you remove or shrink the largest member, the remaining largest member governs the trailing padding requirement.

Signed Numbers & Two's Complement

high yield

Two's Complement Formula

  • Negative representation: -x = ~x + 1 (flip all bits, add 1)
  • Key property: -x + x = 0 (addition wraps mod 2N)
  • MSB (most significant bit) = sign bit: 1 → negative, 0 → non-negative
  • Same hardware circuits handle signed and unsigned addition
Value8-bit two's complement
00x00 = 0000 0000
10x01 = 0000 0001
127 (max+)0x7f = 0111 1111
-10xff = 1111 1111
-128 (min)0x80 = 1000 0000

Converting Signed Hex → Decimal (3-Step Method)

If the leading hex digit is 8–F, the number is negative. Use flip+add-1:

Convert 0xFFFFFFEC (32-bit signed int):

Step 1: Leading digit F → 1111 → MSB=1 → NEGATIVE

Step 2: Flip all hex digits (subtract each from F):
  FFFFFFEC → ~FFFFFFEC = 00000013

Step 3: Add 1, then negate:
  0x00000013 + 1 = 0x00000014 = 20
  Final answer: -20

Shortcut table (leading digit → sign):
  0–7 → positive (read directly)
  8–F → negative (flip+add-1)

⚠ Arithmetic vs Logical Right Shift

The >> operator behaves differently depending on signed vs unsigned:

// UNSIGNED (logical shift): fills left with ZEROS
unsigned int x = 0xF0000000;
x >> 3; // = 0x1E000000 (fills with 0s) ✓

// SIGNED (arithmetic shift): fills left with SIGN BIT
int y = 0xF0000000; // negative! MSB=1
y >> 3; // = 0xFE000000 (fills with 1s!) ✗ WRONG for bit extraction

F25 Snake exam: int food_pts = c >> 3 is wrong because if MSB is set, sign extension corrupts the result. Fix: cast to (unsigned int) first.

Overflow & Undefined Behavior

  • Signed integer overflow = undefined behavior (UB)
  • Unsigned overflow = well-defined: wraps mod 2N
  • INT_MAX + 1 → UB; optimizer may remove code that relies on it
  • 0x7fff'ffff + 1 = 0x8000'0000 = the most negative int (in practice, UB)
  • UBSan: compile with -fsanitize=undefined to detect at runtime
  • 0xffff'fffe as signed int = -2 (leading F → negative, flip+add-1: 0x01+1=0x02=2)
TWO'S COMPLEMENT: Why -1 = 0xFF (for 1 byte / 8 bits)
  0x01  =  0000 0001   (+1)
+ 0xFF  =  1111 1111   (-1 in two's complement)
─────────────────────
  0x100 → carry bit dropped → 0x00 = 0  ✓

To find -x from x:   flip all bits (~x), then add 1
Example: x = 5 = 0000 0101
  ~x    =       1111 1010
  ~x+1  =       1111 1011  = 0xFB = -5  ✓  (check: 5 + (-5) = 0)
⚠ Little-endian trap: On x86-64, multi-byte integers are stored in memory with the least significant byte first. So the integer 0x0000'0001 is stored in memory as bytes 01 00 00 00. When reading hex dumps, remember to reverse byte order to get the integer value.

Spotting Negatives in Hex Dumps

  • Any value with MSB = 1 is negative when signed
  • For a byte: 0x80–0xFF are negative
  • For a 4-byte int: 0x8000'0000–0xFFFF'FFFF are negative
  • 0xFF as unsigned char = 255; as signed char = -1
  • Same bits, different interpretation — classic exam question

Reading Little-Endian Hexdumps (Step-by-Step)

x86-64 stores integers LSB-first. To read an int from a hexdump, reverse the bytes then interpret:

// 4-byte int (positive):
Hexdump:  0C 00 00 00
Reversed: 00 00 00 0C  =  0x0000000C  =  12

// 4-byte int (negative):
Hexdump:  FC FF FF FF
Reversed: FF FF FF FC  =  0xFFFFFFFC  → leading F = negative
Flip+add-1: ~FFFFFFFC = 00000003, +1 = 4 → answer: -4

// 8-byte pointer:
Hexdump:  80 69 70 c6 fc 7f 00 00
Reversed: 00 00 7f fc c6 70 69 80  =  0x00007ffcc6706980
// near 0x7fff → STACK address

// Why 2 hex digits = 1 byte:
// 1 hex digit = 4 bits (nibble), so 2 × 4 = 8 bits = 1 byte
// "0C" = 0000(0) 1100(C) = 0000_1100 in binary

Signed Hex → Decimal (Flip-and-Add-1)

Leading digit 0–7 = positive (read directly). Leading digit 8–F = negative:

// Convert 0xFFFFFFEC:
Step 1: F is 1111 in binary → MSB=1 → NEGATIVE
Step 2: Flip (subtract each hex digit from F):
        ~FFFFFFEC = 00000013

// Common shortcuts to memorize:

Caching & Storage Hierarchy

conceptual + perf

Definition

A cache is a small amount of fast storage used to speed up access to larger, slower storage. Each layer of the hierarchy acts as a cache for the layer below it.

STORAGE HIERARCHY — faster/smaller/costlier at top, slower/larger/cheaper at bottom
Registers
~300 bytes · 0.5 ns · on-chip
↕
L1 / L2 / L3 Cache (SRAM)
KB–MB · ~5 ns · on-chip
↕
DRAM (Main Memory)
8 GB · 60 ns · separate chip
↕
SSD / Flash
512 GB · 60 µs · 1000× slower than DRAM
↕
Hard Disk (HDD)
2–5 TB · 4–13 ms · 100,000× slower than DRAM

Cache Performance

  • Cache hit: data already in cache → fast access
  • Cache miss: must fetch from slower storage below
  • Sequential access → ~74% LLC hit rate (prefetching helps)
  • Random access → ~22% LLC hit rate → 16× slower
  • Locality of reference = accessing nearby addresses in sequence → maximizes hit rate

Why Alignment ↔ Caching

  • CPU fetches memory in fixed-size cache lines (blocks), not individual bytes
  • An unaligned int could span two cache lines → requires two fetches
  • Aligned data always fits in one cache line → one fetch
  • This is the physical reason alignment rules exist
Key latency numbers to know: DRAM ≈ 60 ns · SSD ≈ 60 µs (1000× slower) · HDD ≈ 10 ms (100,000× slower). A system call costs ~1–2 µs = ~20× a DRAM access.

File I/O & File Descriptors

systems concepts

File Descriptors (FDs)

  • FDs are integer handles the kernel uses to track open resources per-process
  • FD 0 = stdin, FD 1 = stdout, FD 2 = stderr — always pre-allocated
  • First file you open() gets FD 3 (not 0)
  • open() → allocates FD; close() → releases it
  • Kernel maintains offset (current read/write position) per FD; use lseek() to change it

read() / write() API

int read(int fd, void* buf, size_t count)
  → returns bytes read, 0 at EOF, -1 on error

int write(int fd, void* buf, size_t count)
  → returns bytes written, -1 on error
  • Kernel advances FD offset by bytes read/written automatically
  • System calls cost ~1–2 µs each — avoid in tight loops
  • O_SYNC flag forces immediate disk write → very slow (100s bytes/sec)
  • Without O_SYNC: writes go to OS page cache first → hundreds MB/sec
⚠ I/O caching risk: Without O_SYNC, a successful write() does not guarantee data is on disk. If power is lost before the OS flushes its page cache, data is gone even though write() returned success.
ℹ Why I/O must go through the OS: The OS mediates all I/O to ensure only one process at a time modifies physical hardware (disk, network), enforces permissions, and enables fair sharing. User-space programs cannot access hardware directly (except via special memory-mapped I/O).

Assembly Language & Registers

high yield

Three Instruction Types

  1. Computation — operates on register values (e.g., addq %rax, %rbx → %rbx := %rbx + %rax). Source operand first, destination second.
  2. Data movement — moves data between registers and memory (e.g., movq %rax, %rbx copies %rax → %rbx). Source first, destination second.
  3. Control flow — changes %rip, the instruction pointer (branches, calls, returns).
⚠ AT&T vs Intel syntax: We use AT&T (percent signs on registers, source before destination). Intel syntax reverses the order — destination first. Don't mix them up reading outside sources.

Calling Convention & The Stack

high yield

x86-64 Linux Calling Convention

  • Args 1–6 → registers: %rdi, %rsi, %rdx, %rcx, %r8, %r9
  • Args 7+ → pushed onto caller's stack frame
  • Return value → %rax
  • Caller-saved registers: callee may overwrite freely (%rax, %rdi, %rsi, %rdx, %rcx, %r8–%r11, %rflags)
  • Callee-saved registers: callee must restore before returning (%rbx, %rbp, %rsp, %r12–%r15)

⚠ What Happens When f() Calls g()

f() was the callee (working for main). Now f() becomes the caller of g(). The same 6 argument registers are reused — if f() still needs its own args, it must save them first:

  • %rdi etc. are caller-saved — g() is allowed to destroy them
  • f() must pushq %rdi before calling g() if it still needs its original arg 1
  • After g() returns, f() does popq %rdi to restore
  • If f() is done with an arg, it can skip the save/restore — no need to waste instructions

callq / retq Mechanics

  • callq FUNC ≡ pushq $NEXT_INSTR; jmp FUNC — pushes return address then jumps
  • retq ≡ popq %rip — pops return address and jumps to it
  • Return address is stored in caller's stack frame (not callee's!)
  • pushq %rax ≡ subq $8, %rsp; movq %rax, (%rsp)
  • popq %rax ≡ movq (%rsp), %rax; addq $8, %rsp
⚠ Common misconception: return address is in the caller's frame, not the callee's. It sits just above the callee's entry %rsp.

The Red Zone — Why Locals Can Be Below %rsp

In unoptimized small functions, you may see args stored at -4(%rsp) without any subq to move %rsp first. This is the Red Zone:

  • x86-64 Linux guarantees the 128 bytes below %rsp won't be overwritten by signals or interrupts
  • Compiler shortcut: tiny leaf functions (no further calls) can skip adjusting %rsp entirely
  • The data is conceptually inside f()'s stack frame even though it's physically below %rsp
  • Normally (non-leaf functions): must subq $N, %rsp to officially claim the space

Base Pointer (%rbp) Convention

  • Callee entry: pushq %rbp then movq %rsp, %rbp
  • %rbp = stable reference to callee's frame base for whole function duration
  • (%rbp) = saved caller's %rbp ← what stored %rbp POINTS TO
  • 8(%rbp) = return address ← 8 bytes above stored %rbp (higher address)
  • This chain of saved %rbp values enables stack unwinding (GDB backtraces, ASan errors)
  • Callee exit: movq %rbp, %rsp; popq %rbp; retq (= leave; retq)

⚠ What Stored %rbp Points To — The Exam Question

When GDB prints a backtrace, it follows this chain: current %rbp → (%rbp) → the value at that address → etc. This is "stack unwinding."

Buffer Overflow Attacks

  • If a function writes beyond the end of a stack buffer, it overwrites the return address
  • Attacker can replace return address with address of any function in the binary
  • Attack payload size = buffer size + saved %rbp + return addr
  • Stack frames on x86-64 Linux are aligned to 16-byte boundaries

Buffer Overflow — Direction of Overflow

Stack grows down (to lower addresses), but array indexing goes up (to higher addresses). So overflowing a buffer writes toward higher addresses — toward saved %rbp and then the return address:

Return address ← overwritten last (target)
Saved %rbp ← overwritten second (8 bytes of junk)
char buf[N] ← overflow starts here, writes UP ↑
↑ lower addresses (where %rsp points)

Modern defenses: stack canaries (detect if buf overwritten), ASLR (randomize addresses), NX bit (non-executable stack)

Full call/return sequence: caller saves caller-saved regs → stores args in %rdi/%rsi/… → callq (pushes return addr + jumps) → callee pushq %rbp → movq %rsp,%rbp → does work → result in %rax → leave; retq → caller restores.

High-Yield Traps & Edge Cases

exam traps

🔢 Number System Traps

  • 0x41 = decimal 65 = ASCII 'A' — hex and ASCII collide constantly
  • Binary has no standard prefix in C (0b is a GCC extension, not ISO C)
  • 0 prefix = octal in C: 010 = 8, not 10
  • 210 = 1024 ≈ 1000, but not equal; use GiB/MiB when you mean powers of 2

🧵 String Traps

  • "Hello" = 5 chars but takes 6 bytes due to NUL
  • '0' (char) = 48 ≠ 0 (integer/NUL)
  • char* = pointer to first char; does NOT store the string itself
  • Assigning p = "literal" makes p point to static memory — not a copy

📍 Pointer Traps

  • All pointer types are 8 bytes on 64-bit regardless of pointed-to type
  • & gives address; * follows address — don't swap them
  • int* p, q; — q is int, not int*
  • Uninitialized pointer = garbage address → dereferencing it = undefined behavior

📐 Alignment Traps

  • Struct size ≠ sum of member sizes due to padding
  • Reordering struct members in source can reduce size; compiler cannot do it for you
  • sizeof(struct x) includes padding — use it instead of manual math
  • Trailing padding is added so arrays of the struct stay aligned

🔣 Two's Complement Traps

  • 0xFF as unsigned char = 255; as signed char = -1
  • Signed overflow = undefined behavior; optimizer can delete your code
  • Unsigned overflow = well-defined wraparound (not UB)
  • Little-endian: bytes in memory are reversed from how we write numbers

🗂 Memory Segment Traps

  • Stack grows down; heap grows up — they collide if memory runs out
  • Heap can be fragmented (holes); stack cannot
  • Returning pointer to local = dangling pointer (local freed at })
  • malloc returns void*; always cast to the correct type

🖥 Assembly Traps

  • AT&T syntax: source first, destination second — Intel is reversed
  • asm l suffix = 32-bit; C's long = 64-bit (asm q suffix)
  • Registers have no memory addresses — cannot take &%rax
  • Backward branch target = almost always a loop

📞 Calling Convention Traps

  • Return address stored in caller's stack frame, not callee's
  • Args 7+ go on caller's stack, not registers
  • Callee-saved: %rbx, %rbp, %rsp, %r12–%r15 — callee must restore these
  • Buffer overflow overwrites the return address sitting above the overrun buffer

Project 1: Snake

applied concepts

What It Is

You implemented the classic Snake game in C — a moving snake on a grid, eating food, growing via a linked list, and dying on wall/self collisions. The project was split into parts that tracked lecture topics: pointers → bitwise ops → strings → linked lists → structs → Unicode.

Bit Flags for Cell State

Each board cell is a single int with individual bits encoding its properties simultaneously:

#define PLAIN_CELL  0b0000  // 0
#define FLAG_SNAKE  0b0001  // 1
#define FLAG_WALL   0b0010  // 2
#define FLAG_FOOD   0b0100  // 4
#define FLAG_GRASS  0b1000  // 8
  • Set a flag: cell |= FLAG_SNAKE
  • Clear a flag: cell &= ~FLAG_SNAKE
  • Check a flag: if (cell & FLAG_WALL)
  • Snake on grass = FLAG_SNAKE | FLAG_GRASS = 0b1001 = 9
  • When snake leaves: cell &= ~FLAG_SNAKE — only removes snake bit, preserves grass

Global Variables + extern

  • Globals defined in .c file, declared with extern in .h file
  • extern doesn't create a new variable — it tells other files one exists
  • Two-step: define in common.c, declare extern in common.h
  • Without extern, each file would get its own copy → different values
// common.c — definition (storage lives here)
int g_game_over;
int g_score;

// common.h — declaration (tells other files it exists)
extern int g_game_over;
extern int g_score;

Structs for Snake Data

  • Part 3 moved snake data from scattered globals into a single snake_t struct
  • Struct access: snake.head_row (value) vs snake_p->head_row (pointer)
  • Struct passed by pointer to functions so they can modify it in-place
  • Linked list used for snake body: head at front, tail at back; add to front on move, remove from tail (unless growing)

Linked List for Snake Body

  • Linked list = sequence of node_t structs, each holding coordinates + pointer to next
  • On each move: prepend new head position, remove tail unless eating food
  • Self-referential struct: struct node* next inside struct node — uses raw struct node* not node_t* because typedef isn't defined yet
  • Cannot embed struct node itself (infinite size); only a pointer to it

Memory Management in Snake

  • cells array is heap-allocated by malloc() — must be freed with free(cells)
  • LeakSanitizer reports if malloc'd memory is never freed
  • Leak = program exits with still-allocated heap memory
  • Fix: call free(cells) in end_game() before exit
  • Linked list nodes also need freeing when game ends — each node allocated with malloc()

UTF-8 / Unicode (mbslen)

  • ASCII = 1 byte/char, only 128 chars; doesn't cover non-Latin alphabets or emoji
  • UTF-8 = variable width: 1–4 bytes per character
  • Leading bits encode length: 0xxxxxxx=1B, 110xxxxx=2B, 1110xxxx=3B, 11110xxx=4B
  • Continuation bytes always start with 10xxxxxx
  • strlen() counts bytes, not characters — use mbslen() for UTF-8 character count
  • Check if byte is a start byte: (byte & 0b11000000) != 0b10000000 (i.e., not a continuation byte)
⚠ Game loop trap: Initialize globals in initialize_game(), NOT in main() — the autograder calls initialize_game() directly and expects globals set there. Also: the snake should not move into a wall — detect the collision before updating position, and set g_game_over = 1.
Key bitwise idioms: set = OR (|=), clear = AND with NOT (&= ~FLAG), test = AND (& FLAG). XOR (^=) toggles a bit. These patterns appear on exams directly.

Project 2: DMalloc — Debugging Memory Allocator

high yield

What It Is

You built a drop-in replacement for malloc()/free() that tracks every allocation and detects memory bugs: double frees, invalid frees, buffer overflows, and leaks. Every call to dmalloc(sz, file, line) allocates slightly more memory than requested and stores metadata around the user's block.

What dmalloc Tracks

  • nactive: current live allocations (#malloc − #free)
  • active_size: total bytes in live allocations
  • ntotal / total_size: all allocations ever made
  • nfail / fail_size: failed allocation attempts (e.g., size too large)
  • heap_min / heap_max: smallest and largest addresses ever allocated

Metadata Layout

Each allocation stores metadata adjacent to the user's block. A typical layout:

Header
size, file, line, magic
User data
what malloc() returns a pointer to
Footer
"canary" bytes
  • dmalloc(sz) calls base_malloc(header_size + sz + footer_size)
  • Returns pointer to user data region, not to header
  • On free: walk back to header using pointer arithmetic

Error Detection Methods

ErrorHow detected
Double freeCheck a "is_freed" flag in header; if already 1 → error
Invalid free (not heap)Check if ptr is within [heap_min, heap_max]
Invalid free (not start)Walk active allocation list; if ptr is inside a block → "X bytes inside a Y byte region"

Integer Overflow Protection

  • malloc(sz) parameter is size_t — an unsigned type that wraps on overflow
  • In calloc(n, size): if n * size overflows, the computed size wraps to a small number and malloc succeeds with a dangerously small block
  • Fix: check n > SIZE_MAX / size before multiplying — if true, return NULL
  • Also check: sz == 0 edge case (legal; returns non-NULL per C++ standard)
  • malloc(0) returns non-null; free(nullptr) is always legal (does nothing)

base_malloc vs malloc

  • base_malloc/base_free are custom wrappers — they don't modify freed memory, and never return freed memory to the OS
  • This makes behavior reproducible and avoids the real malloc's undefined behavior when testing double-free etc.
  • Debugging allocators need these guarantees to reliably detect bugs — real malloc's behavior after UB is unpredictable

Canary / Boundary Write Detection

  • Write a known "magic" byte pattern into the 8 bytes after the user's allocation
  • On free(), re-read those bytes — if they changed, a write overflowed the buffer
  • Can only detect writes within ~8 bytes of the end (not arbitrarily far away)
  • Error message format: MEMORY BUG: file:line: invalid free of pointer 0x…, not allocated
⚠ Output format is exact. DMalloc tests compare your output character-by-character. One extra space or a missing newline fails the test. When a pointer is inside an existing block: print both the invalid-free message and the "X bytes inside a Y byte region allocated here" line on the next line.
malloc rule: always returns 16-byte aligned pointer on x86-64. sizeof is compile-time only. free(NULL) is legal and does nothing. double-free and use-after-free are undefined behavior — UBSan (-fsanitize=undefined) catches them at runtime.

Project 3: File I/O Cache

applied concepts

What It Is

You implemented a caching I/O library — a layer between your program and the OS that buffers file reads/writes in memory (DRAM) to avoid slow disk syscalls. The goal: make file operations as fast as the standard library's stdio, which also caches internally. Without caching, byte_cat on a 1 MiB file took ~18 seconds; with caching, ~0.14 seconds — a ~100× speedup.

The Core API

FunctionWhat it does
io300_readc(f)Read 1 byte from cache (fetch from disk if needed)
io300_writec(f, ch)Write 1 byte to cache (flush to disk when full)
io300_read(f, buf, sz)Read sz bytes into buf — NOT just repeated readc calls
io300_write(f, buf, sz)Write sz bytes from buf — NOT just repeated writec calls
io300_seek(f, pos)Move to byte offset pos in file (may require flush)

read must NOT call readc in a loop — same for write/writec. Doing so defeats caching.

Key Metadata to Track

  • Cache buffer: fixed-size byte array (e.g., 8 bytes in tests)
  • File offset: where in the file the cache currently corresponds to (which chunk is loaded)
  • Cache position: current read/write position within the cache buffer
  • Cache valid bytes: how many bytes in the cache are actually from the file (last chunk may be partial)
  • Dirty flag: whether the cache has been written to but not yet flushed to disk

Fetch vs Flush

  • Fetch: read a chunk from disk into the cache (happens when cache miss on read)
  • Flush: write dirty cache contents back to disk (happens before fetch, on seek, or on close)
  • Always flush before fetch — otherwise you'd overwrite unwritten data
  • On seek(): if new position is outside current cached chunk → flush if dirty, then fetch new chunk
  • On close(): always flush if dirty

Why System Calls Are Expensive

  • Each read()/write() syscall costs ~1–2 µs
  • Naive implementation: one syscall per byte → millions of syscalls for a 1 MB file
  • Caching reduces this to one syscall per cache-sized chunk → orders of magnitude fewer calls
  • Syscall cost = switch to OS mode + safety checks + hardware I/O — much more than a function call
  • strace -e trace=read ./program shows every read syscall — use this to verify caching

Read/Write Mixed Operations Trap

  • The hardest case: alternating reads and writes on the same file
  • After a write: cache is dirty; if a subsequent read needs a different chunk, you must flush first
  • After a read: if a subsequent write targets a different chunk, fetch that chunk first (so you don't overwrite unread data), then write into it
  • Invariant: cache must always reflect a coherent, contiguous region of the file

File Descriptors in Practice

  • Your struct io300_file wraps a file descriptor (int FD) from open()
  • All actual disk I/O uses read(fd, buf, n) and write(fd, buf, n) syscalls
  • lseek(fd, offset, SEEK_SET) moves the kernel's FD offset to offset
  • You must track the kernel offset yourself — it advances automatically after each read/write
  • On close: flush dirty cache, then close(fd)
ℹ Design principle: The cache stores one contiguous aligned chunk of the file at a time. On a cache miss, flush (if dirty) then fetch the new chunk. All readc/writec/read/write calls operate on the cache; only fetch/flush touch the disk. This is the same principle that CPU caches use for RAM — just at the software level.
⚠ Performance gotchas: (1) O_SYNC flag forces every write to hit disk immediately — extremely slow. Without it, writes go to the OS page cache first. (2) Your io300_read must copy a whole chunk in one read() syscall, not call io300_readc in a loop. (3) Writes past EOF extend the file — handle this; don't assume file size is fixed.
The three-layer hierarchy in P3: your cache (user-space DRAM) → OS page cache (kernel DRAM) → disk (SSD/HDD). Even without O_SYNC, two caching layers are at play. The OS may lose unflushed page cache data on power failure even after a "successful" write().

struct { int i1; int i2; int i3; char c1; char c2; char c3; }

  • C!! question (S24): struct member order cannot be reoptimized by compiler — so the first member rule ((T*)struct == &struct.first_member) is the only reliable one

⚠ Signed vs unsigned overflow — the most-tested distinction: int i2 = 0x7FFF'FFF0; i2 + 0x64 overflows → UB. But unsigned long l = UINT64_MAX; l + 1 → 0, well-defined. The compiler can assume signed overflow never happens and will delete branches that rely on it.

Stack Diagram Traps (every exam)

  • Return address is in caller's frame — every exam has someone put it in the wrong frame
  • After retq: %rsp advances by 8 (the return address is popped), ending up pointing at the return address's old position + 8
  • Stored %rbp points to caller's %rbp — so it's an address within the caller's frame
  • Stack grows down: locals at -4(%rbp), -8(%rbp) etc. — lower addresses
  • If function has returned, its whole frame is gone — the stack pointer is back at caller's entry %rsp
  • S24 trap: diagram showed mystery()'s frame missing — correct, it had already returned!

Buffer Overflow Attack Math (S24)

Given a vulnerable function with a local buffer, calculate overflow payload size:

sub $0x20, %rsp        ← 32 bytes local space
lea -0x20(%rbp), %rax  ← buffer starts 32B below %rbp
movl $0, -0x4(%rbp)    ← 4B local int at -4(%rbp)

Payload = buffer_size + saved_%rbp + return_addr
        = 32 bytes   +  8 bytes   +  8 bytes  = 48

Note: stack frames must be 16-byte aligned, so padding may be needed. The exam answer was 44 because the buffer was at -0x20 from %rbp (32B) plus 4B to the saved %rbp = 36B to reach the return address, plus 8B for the address itself = 44.

⑤ Assembly Reverse-Engineering

Every exam asks you to reconstruct C from assembly. The standard approach:

Step-by-Step Method

  1. Identify arguments from stack comments or register use (%edi=arg1, %rsi=arg2, etc.)
  2. Find all local vars by their stack offsets (-4(%rbp), -8(%rbp), etc.)
  3. Find the loop: look for a backward branch (jXX .L_earlier)
  4. Identify loop variable by what gets incremented and compared
  5. Read loop body between the label and the conditional branch at the end
  6. Work backwards from ret: what's in %eax/%rax just before it?

⑥ Bitflags — Snake/Serpent Questions (S24, S25, F25)

What the Exam Tests

  • Flags must be powers of 2 (each occupies exactly one bit) to be OR-combinable
  • If flags are 1,2,3,4 (not powers of 2) — bitwise tests break. cell & FLAG_WALL where FLAG_WALL=3 would match any cell with bits 0 or 1 set
  • Testing "exactly snake AND food and nothing else": cell == (FLAG_SNAKE | FLAG_FOOD) = cell == 0x05
  • Testing "has snake AND has food (may have other flags)": (cell & FLAG_SNAKE) && (cell & FLAG_FOOD)
  • Moving into cell: use |= not += — addition would corrupt if the bit was already set
  • XOR toggle (^=) is wrong for moving into a cell — would remove the flag if already set

Enum Pitfall (S24, S25)

Using sequential enum values (0,1,2,3,4) instead of bit flags breaks bitwise operations:

// WRONG: these are NOT bit flags
enum { LAND=0, SNAKE=1, WATER=2, WALL=3, FOOD=4 }

cell = SNAKE | WATER; // = 1 | 2 = 3 = WALL!
(cell & SNAKE) // = 3 & 1 = 1 ✓ but...
(cell & WALL)  // = 3 & 3 = 3 ✓ FALSE POSITIVE

// CORRECT: powers of 2
#define FLAG_SNAKE 0b0001
#define FLAG_WATER 0b0010
cell = FLAG_SNAKE | FLAG_WATER; // = 0b0011
(cell & FLAG_SNAKE) // = 1, truthy ✓
(cell & FLAG_WALL)  // = 0b0011 & 0b0100 = 0 ✓

⑦ Hexdump Reading & Signed Number Interpretation

Reading a Hexdump (S25 Q8, S24 Q1B)

Given a hexdump, reconstruct int values. Remember: little-endian — reverse each 4-byte group, then flip+add-1 if leading digit is 8–F.

// Full worked example (S25 Q8) — array of int:
0x4af45ab0: 0C 00 00 00  10 00 00 00  FC FF FF FF  FF FF FF FF
0x4af45ac0: A1 00 00 00  FA FF FF FF  FF FF FF 7F  FF AB 02 00

arr[0]: 0C 00 00 00 → 12      arr[4]: A1 00 00 00 → 161
arr[1]: 10 00 00 00 → 16      arr[5]: FA FF FF FF → -6
arr[2]: FC FF FF FF → -4      arr[6]: FF FF FF 7F → INT_MAX
arr[3]: FF FF FF FF → -1      arr[7]: FF AB 02 00 → 175,103

// Count: 8 ints total (32 bytes / 4 bytes each)

// arr[0] | arr[4]:  0x0C | 0xA1
// 0000_1100 | 1010_0001 = 1010_1101 = 0xAD = 173

// arr[0] + arr[6]:  12 + 2,147,483,647 → signed overflow → UB!
// arr[2] = -4 in decimal (FC FF FF FF → flip+1)
// arr[3] + arr[0]: -1 + 12 = 11 (no overflow)
  • Each pair of hex digits = 1 byte (1 digit = 4 bits = nibble)
  • Stack pointer near 0x7fff..., heap near 0x1a0... or 0x55a...
  • Many 00 bytes after a value = small positive int (leading zeros)

Overflow Arithmetic — Exact Questions (F25, S23)

The exam asks: does this expression overflow? If signed, is it UB?

// S23 examples:
add(1, 0xFFFF'FFFA)
// = 1 + (-6) as signed = -5. NO overflow, no UB.

add(0x7FFF'FFAC, 0x64)
// = INT_MAX-ish + 100. YES overflow → UB!

sub(300, 330)  // unsigned: 300-330 wraps. YES overflow, NO UB.

// F25: add(0xffff'fffe, ???) result 3
// 0xffff'fffe = -2 as signed int
// -2 + 5 = 3. Answer: add(0xffff'fffe, 5)

Key: 0xFFFF'FFFF as signed int = -1. 0x8000'0000 = INT_MIN = -2147483648. 0x7FFF'FFFF = INT_MAX.

Pointer Arithmetic to/from Header (S25 Q5A)

Given void* ptr pointing to the payload, recover the header:

// Header is sizeof(metadata) bytes BEFORE ptr
// Must cast to char* first to do byte arithmetic!

// CORRECT:
struct metadata* m = (struct metadata*)((char*)ptr - sizeof(struct metadata));
// Also correct:
struct metadata* m = ((struct metadata*)ptr) - 1; // pointer arith: -1 × sizeof(metadata)

// WRONG — advances by sizeof(int) not sizeof(metadata):
struct metadata* m = ((int*)ptr) - sizeof(int);
⚠ Pointer arithmetic scales by sizeof(*ptr). ((struct metadata*)ptr) - 1 goes back exactly sizeof(struct metadata) bytes — correct. ptr - 1 on a void* is illegal in C (undefined). Always cast to char* for raw byte arithmetic.

Metadata Placement: Before vs After (S24 Q8B)

  • Standard: metadata before payload → boundary write past end corrupts next block's header (detectable)
  • Joel's design (S24): metadata after payload → boundary write past end corrupts own metadata → two failure cases: (1) write goes past the canary range entirely, (2) write corrupts only the freed flag, making freed appear as 0 (valid)
  • Key: if canary is between payload and metadata, a write that misses the canary but hits the freed flag can make a freed block look allocated

Canary Detection Math (F25 Q5A)

Given addresses of allocations and 8-byte canaries, determine if a write is detectable:

  • a[65] where a=64B → writes at byte 65 of payload, canary is at bytes 64–71. Byte 65 is within canary → detectable
  • *(b - 200) → 200B before b's start = before the canary of the previous block's end → not detectable by b's canary; may not align with a's canary either
  • *(b + 0x344) = b + 836B; b is at 0x200, canary ends at 0x248 (64+8=72B total); 836 >> 72 → not detectable
  • Rule: canary only covers the 8 bytes immediately after the payload

⑨ Caching I/O — Cache Hit Rate Questions

Hit Rate by Access Pattern (S23 Q7B)

ProgramCache sizeHit rateWhy
byte_cat 4MB4096B~99.97%Sequential: 1 miss per 4096 bytes = 1/4096 miss rate
block_cat 512 4MB4096B~87.5%512B blocks: 1 miss then 7 hits per 4096B chunk
reverse_byte_cat 4MB4096B~0.02%Each byte seek = new cache miss; going backward fetches from new position each time
stride_cat 1024 512 1MB4096B~0%Stride 1024 > cache 4096/4 slots — each read misses; alternating between non-overlapping regions

Formula: hit rate = (cache_size - 1) / cache_size for sequential byte reads. One miss per cache-fill, rest are hits.

Seek Strategy Tradeoffs (S24 Q5A, F25 Q6A)

  • Kazuya (aligned blocks): fetches fixed 64-byte aligned blocks. reverse_byte_cat seeks backward 1 byte at a time → each seek within same aligned block = cache hit (up to 64 consecutive seeks hit)
  • Alex (seek-to-target): fetches from exact seek position. reverse_byte_cat → each seek fetches 1 new block, immediately seeks past it → constant misses
  • Cash (direction-aware): pre-fetches in the direction of movement. reverse_block_cat = backward → fetches 8B before seek target → more hits for backward reads
  • Rule: aligned-block caching beats seek-to-target for reverse access because each backward step usually stays in the same block
